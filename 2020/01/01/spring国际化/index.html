-spring -国际化

1. spring 国际化 MessageSource

定义 MessageSource 接口，标识支持国际化的获取资源的方法，Spring 中对 MessageSource 提供了三个实现类，分别是 ReloadableResourceBundleMessageSource、StaticMessageSource 和 ResourceBundleMessageSource。

实现类 作用
ReloadableResourceBundleMessageSource 带有缓存和过期时间的，可重新加载的 MessageSource实现类
ResourceBundleMessageSource 缓存的，以BundleMessage实现的MessageSource

2. ReloadableResourceBundleMessageSource

类结构

  • HierarchicalMessageSource 该接口描述了带有层次结构的 MessageSource
  • ResourceLoaderAware 该接口描述了实现类能够被注入 ResourceLoader
  • AbstractMessageSource 主要实现了层次结构，CommonMessage, 和 getMessage 方法，在其实现的 getMessage 方法中主要实现了一些参数填充，决策是否采用 default 的逻辑等，而抽象了通过 code 和 locale 获取 MessageFormat 的方法，由其子类实现。
  • AbstractResourceBasedMessageSource 定义缓存和加载资源名字等。

类结构图

ReloadableResourceBundleMessageSource 通过内部类 PropertiesHolder 缓存数据，同时支持线程安全加载资源，其线程安全的加载资源实现形式是通过 ConcurrentHashMap 结合PropertiesHolder内部的 ReentrantLock 来实现的。具体代码如下：

  • 通过刷新的时间戳 refreshTimestamp 判断是否已经过了 cache 的时间，从而进行更新
  • concurrentRefresh 字段用于配置是否允许并发刷新，如果允许并发刷新，那么在需要刷新并且不是全新的没有注入数据的 Holder 的时候没有获取成功锁, 证明别的线程在刷新，当前线程直接返回，不再去刷新数据；如果不允许并发刷新，那么就根据获取锁的顺序依次刷新数据。
  • 这里在并发刷新的时候，如果是全新的 Holder 那么线程会获取锁并刷新数据后再返回
  • 通过以上两点来保证线程安全和数据安全
  • 在获取到锁之后还会再次从 map 中获取最新的 Holder 并对比刷新时间，如果刷新时间大于之前获取的 Holder 的刷新时间，那么证明该 Holder 是别的线程刷新完成并放入的最新的 Holder，此时便没有必要再次刷新了，直接返回。这里为什么说对比刷新时间就能得知是否是已经刷新过的，并且是数据完全的不会 NPE 的呢，是因为 Holder 里面的 refreshTime 是 volatile 的，并且是在数据处理并加载到 map 完成后最后一步才更新的 refreshTime，因此可以保证线程之间的可见性。
  • ReloadableResourceBundleMessageSource 中 properties 文件的读取和国际化都是类内部自己实现的，包括了计算文件名等。

3. ResourceBundleMessageSource

类结构图

从类结构图可以看出其也是带有层次结构的 MessageSource，但不是从 ResourceLoader 加载资源了。另外，该类的国际化资源加载使用的是 java 的 ResourceBundle 来实现。并且不支持缓存失效重新加载。所以在该类进行设置缓存时间无效。
